技术选型分析

多种技术方案的优劣对比和推荐，结合场景给出有理有据的选型建议。

单体架构 (Monolithic)

优点：

  • 开发简单，部署方便，一个包搞定
  • 调试容易，数据一致性好管理
  • 小团队效率高，学习成本低

缺点：

  • 代码耦合度高，模块边界模糊
  • 扩展困难，只能整体扩容
  • 技术栈固定，升级影响全局

适用场景：早期项目、小型团队、MVP 验证、业务逻辑简单的系统

微服务架构 (Microservices)

优点：

  • 服务独立部署，故障隔离好
  • 技术栈灵活，各服务可用不同语言
  • 水平扩展灵活，按需扩缩容

缺点：

  • 运维复杂度高，需要完善的基础设施
  • 分布式事务处理困难
  • 服务间通信延迟，调试链路复杂

适用场景：中大型项目、多团队协作、高可用要求、业务复杂且需要独立演进

Serverless 架构

优点：

  • 按需付费，无需管理服务器
  • 自动弹性伸缩，应对流量波动
  • 开发速度快，聚焦业务逻辑

缺点：

  • 冷启动延迟，不适合低延迟场景
  • 厂商锁定，迁移成本高
  • 调试困难，本地开发体验差

适用场景：事件驱动型应用、流量波动大的场景、快速原型开发、定时任务处理

💡 完整实战：电商系统架构设计

用 BUILD 框架一步步完成一个电商系统的架构设计，展示 AI 在每个环节的实际输出。

1

需求分析 — AI 提取技术需求

把产品需求抛给 AI，让它帮你提取技术约束和非功能性需求。

我要设计一个电商系统，支持以下功能：
- 用户注册登录、个人中心
- 商品浏览、搜索、分类筛选
- 购物车、下单、在线支付
- 订单管理、物流追踪
- 商家后台：商品管理、订单处理

预期规模：日活 10 万用户，峰值 QPS 5000

请分析：
1. 核心技术需求
2. 非功能性需求（性能、可用性、安全性）
3. 技术挑战和风险点

🤖 AI 输出示例

核心技术需求：用户认证系统、商品搜索引擎、购物车状态管理、支付网关集成、订单状态机、物流 API 对接

非功能性需求：API 响应 < 200ms（P99）、可用性 99.9%、支付数据加密、SQL 注入防护

风险点：秒杀场景超卖、支付回调一致性、搜索性能瓶颈

2

系统边界 — 定义组件和外部依赖

明确系统包含哪些核心服务模块，以及需要对接的外部系统。

🤖 AI 输出示例

核心模块：

  • 用户服务 — 注册、登录、OAuth、个人信息
  • 商品服务 — 商品 CRUD、分类、库存管理
  • 搜索服务 — 全文搜索、筛选、排序
  • 订单服务 — 下单、状态流转、退款
  • 支付服务 — 支付接入、回调处理、对账

外部依赖：微信/支付宝支付 API、短信服务（阿里云）、对象存储（OSS）、物流查询 API

数据库 Schema — 实体关系与表结构

AI 根据业务模型生成数据库 Schema 和索引策略。

-- 用户表
CREATE TABLE users (
    id          BIGINT PRIMARY KEY AUTO_INCREMENT,
    phone       VARCHAR(20) UNIQUE NOT NULL,
    nickname    VARCHAR(50) NOT NULL,
    avatar_url  VARCHAR(500),
    password    VARCHAR(255) NOT NULL,
    status      TINYINT DEFAULT 1,
    created_at  DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at  DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    INDEX idx_phone (phone)
);

-- 商品表
CREATE TABLE products (
    id            BIGINT PRIMARY KEY AUTO_INCREMENT,
    title         VARCHAR(200) NOT NULL,
    description   TEXT,
    price         DECIMAL(10,2) NOT NULL,
    stock         INT UNSIGNED DEFAULT 0,
    category_id   BIGINT NOT NULL,
    seller_id     BIGINT NOT NULL,
    status        TINYINT DEFAULT 1,
    created_at    DATETIME DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_category (category_id),
    INDEX idx_seller (seller_id),
    INDEX idx_status_created (status, created_at)
);

-- 订单表
CREATE TABLE orders (
    id            BIGINT PRIMARY KEY AUTO_INCREMENT,
    order_no      VARCHAR(32) UNIQUE NOT NULL,
    user_id       BIGINT NOT NULL,
    total_amount  DECIMAL(10,2) NOT NULL,
    status        VARCHAR(20) DEFAULT 'pending',
    paid_at       DATETIME,
    created_at    DATETIME DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_user_status (user_id, status),
    INDEX idx_order_no (order_no)
);

架构评审 — AI 评审设计方案

让 AI 从多个维度评审你的架构设计，发现潜在问题。

请评审以下电商系统架构设计，从以下维度分析：

1. 可扩展性：系统能否支撑 10x 流量增长？
2. 可用性：单点故障在哪里？如何做高可用？
3. 安全性：有哪些安全风险？如何防护？
4. 性能：瓶颈在哪里？如何优化？
5. 可维护性：代码和架构是否便于长期维护？

[粘贴你的架构设计文档]

🤖 AI 评审建议示例

  • 库存超卖风险：建议用 Redis 分布式锁或数据库乐观锁处理并发扣减
  • 搜索性能：商品搜索应接入 Elasticsearch，避免数据库 LIKE 查询
  • 支付一致性：支付回调需幂等设计，建议用消息队列做最终一致性
  • 缓存策略：商品详情页建议加 Redis 缓存，热点商品用本地缓存兜底
  • 安全建议：敏感接口加限流、支付金额服务端校验、SQL 参数化查询

✅ 推荐做法

  • 先用 AI 做需求分析再设计
  • 让 AI 列出多种方案再选择
  • 用 AI 做架构决策记录 (ADR)
  • 让 AI 从不同角度评审设计
  • 迭代优化而非一次完成

❌ 常见误区

  • 完全依赖 AI 做架构决策
  • 不考虑团队能力直接选方案
  • 忽略非功能性需求
  • 跳过性能和安全评审
  • 照搬 AI 建议不做验证
