编写失败的测试

先明确你想要什么——用测试描述期望行为。此时代码尚未实现，测试必须是红色（失败）的。让 AI 帮你根据需求生成完整的测试用例。

📋 每阶段 Prompt 模板

复制即用的 AI 提示词，覆盖 TDD 的三个阶段。

🔴
Red — 编写失败测试
为以下函数签名编写失败测试用例，覆盖正常输入、边界情况和异常输入。使用 Jest/Vitest 框架，每个测试用例包含清晰的描述： function slugify(title: string): string
根据需求文档生成边界测试用例，包括空字符串、超长输入、特殊字符、Unicode 字符等边界情况。确保每个测试独立运行。
🟢
Green — 最小实现
让所有测试通过，使用最简单的实现。不要过度设计，不要提前优化，只需满足当前测试要求即可。
实现以下接口使测试通过。只关注让红色测试变绿，不添加任何测试未要求的功能： interface Slugifier { slugify(input: string): string }
🟡
Refactor — 优雅重构
在保持所有测试通过的前提下重构以下代码。关注点：消除重复、提高可读性、遵循单一职责原则。
提取重复逻辑为独立函数，优化命名，添加类型注解。重构后运行所有测试确保无回归。

⚠️ 常见反模式

这些做法会削弱 TDD 的价值，在使用 AI 辅助时更需警惕。

❌ 先写实现再补测试
先完成功能代码，之后再让 AI 补上测试。这样测试只是验证了已有代码，无法驱动设计。
先向 AI 描述需求，让它生成测试用例，确认测试失败后再实现功能。
❌ 一次性要求所有测试和实现
一条 prompt 让 AI 同时生成全部测试和实现代码，跳过了红-绿-重构的节奏。
分步进行：先让 AI 生成测试，确认失败，再让 AI 写最小实现，最后重构。
❌ 测试太多实现细节
测试紧耦合内部实现（如检查私有方法调用次数），导致重构时大量测试失败。
测试行为而非实现。告诉 AI 只关注输入-输出关系和公共接口。
❌ 跳过重构阶段
测试通过就结束，留下大量"刚好能工作"的代码，技术债务持续累积。
每次绿灯后让 AI 审查代码质量，提出重构建议，持续改善设计。
