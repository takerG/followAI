内联编辑 — 最常用的 AI 交互

入门

内联编辑是 Cursor 最核心的功能之一。选中代码后按下快捷键，直接在编辑器内描述你想要的修改，AI 会生成 diff 供你审核。

打开内联编辑（Edit） CmdK / CtrlK
接受 AI 建议 CmdY / CtrlY
拒绝 AI 建议 CmdN / CtrlN
Pro Tip: 先选中目标代码再按 CmdK，AI 会聚焦在选中区域进行修改，上下文更精确，结果更好。
💡

Tab 智能补全 — AI 的读心术

入门

Cursor 的 Tab 补全远超传统补全，它能预测你接下来要写的多行代码，甚至跨函数补全。

接受补全建议 Tab
部分接受（逐词） Cmd→ / Ctrl→
忽略建议继续输入 Esc
最佳实践: 写一个清晰的函数签名或注释，然后停顿一下——Tab 补全经常能一次性生成完整的实现逻辑。利用逐词接受来精确控制采纳的内容。

入门

聊天面板适合进行较复杂的讨论：理解架构、调试问题、设计方案等。它与编辑器深度集成，可以直接应用代码修改。

打开 / 关闭 AI 聊天面板 CmdL / CtrlL

将选中代码添加到聊天 CmdShiftL（选中后）

聊天面板中 AI 生成的代码块可以一键应用到编辑器，无需手动复制粘贴。

💬 AI 对话技巧

与 AI 沟通的质量直接决定输出质量。掌握以下技巧，让 AI 真正成为你的得力编程助手。

@

@ 符号引用 — 精准提供上下文

进阶

Cursor 的 @ 引用系统让你可以精确指定 AI 应该参考的上下文：

引用特定文件 @filename
引用整个代码库 @Codebase
引用文档/网页 @Docs / @Web
引用 Git 历史 @Git
引用特定函数/类 @symbolName
关键原则: 与其让 AI 猜测上下文，不如主动用 @ 提供精确引用。比如 @utils.ts 参考这个工具函数来实现类似的错误处理——比模糊描述高效得多。
🎯

结构化提示 — 写出好指令

进阶

好的提示 (prompt) 应该包含三个要素：目标、约束、示例。

// 模糊的提示（效果差） 帮我写个表单验证 // 结构化的提示（效果好） 目标：为用户注册表单添加客户端验证 字段：邮箱（RFC 5322）、密码（8位+大小写+数字）、手机（中国大陆） 要求： - 使用 zod schema 定义验证规则 - 实时验证，失去焦点时触发 - 错误提示用中文 - 参考 @form-schema.ts 现有模式

越具体的指令，AI 越能一次性给出满意的结果，减少来回修改。

🔁

迭代对话 — 逐步完善

进阶

复杂功能不要试图一次搞定，使用迭代策略逐步完善：

先搭骨架 — "先创建基本的组件结构和类型定义，不要实现具体逻辑"
填充核心 — "现在实现核心的数据处理逻辑，参考 @api-types.ts 的接口定义"
处理边界 — "添加错误处理、loading 状态和空数据的 fallback"
打磨细节 — "优化性能，添加 useMemo/useCallback，处理竞态条件"
为什么有效: 每一步的上下文都很清晰，AI 更容易给出高质量代码。同时你可以在每一步审查并纠正方向。
📚

上下文窗口管理

高级

AI 模型有上下文长度限制。有效管理上下文窗口可以显著提升回答质量：

及时新建会话：当对话超过 15-20 轮时，AI 对早期上下文的"记忆"会变弱。此时应新建会话并重新提供关键上下文。

精选上下文：不要把整个项目丢给 AI。只引用真正相关的文件（通常 2-5 个），让 AI 聚焦在核心问题上。

利用摘要：长对话中途可以要求 AI "总结目前的设计决策和待办事项"，然后新建会话粘贴摘要继续。

常见误区: @Codebase 虽然强大，但在大型项目中会消耗大量上下文窗口。对于具体问题，精准引用 2-3 个相关文件总是优于全局搜索。
🛠

让 AI 先思考再动手

进阶

对于复杂任务，让 AI 先分析和规划，再执行修改：

// 先分析 分析 @auth-service.ts 的当前认证流程， 列出存在的安全隐患和改进建议， 暂时不要修改代码。 // 确认方案后再执行 按照你建议的方案 2 进行重构， 使用 JWT + Refresh Token 双令牌模式。

这样做有两个好处：你可以在执行前审核方案；AI 有了清晰的计划后，生成的代码质量更高。

调试时的对话策略

进阶

遇到 bug 时，提供完整的调试信息可以大幅提升 AI 的定位效率：

Bug 描述：用户登录后刷新页面，偶尔会被重定向到登录页 复现步骤：登录 -> 等待 3 秒 -> 刷新页面 期望行为：刷新后保持登录状态 实际行为：约 30% 的概率被踢回登录页 相关代码：@auth-middleware.ts @session-store.ts 错误日志：Token validation failed: jwt expired

结构化的 Bug 报告让 AI 可以像有经验的开发者一样系统地排查问题。

多文件重构策略

高级

大规模重构是 Composer 的最佳使用场景。以下是经过实践验证的最佳流程：

列出范围 — 先让 AI 分析："列出从 REST API 迁移到 tRPC 需要修改的所有文件"
分批执行 — 将文件按依赖关系分组，先改底层（类型、工具函数），再改上层（组件、页面）
逐批验证 — 每批修改后运行测试确认无误，再继续下一批
同步更新测试 — 不要把测试留到最后，每步改完代码后立即让 AI 更新对应的测试
避免: 一次让 Composer 修改超过 10 个文件。文件越多，AI 出错的概率越大。宁可多分几次，也不要贪多。

文档与注释的 AI 辅助

入门

.cursorrules — 定义项目级 AI 行为

高级

.cursorrules 文件放在项目根目录，可以全局影响 AI 在该项目中的所有回答。这是提升 AI 输出一致性的最重要手段。

# .cursorrules 示例 你是一位资深全栈工程师。 技术栈： - 前端：React 18 + TypeScript + TailwindCSS + Zustand - 后端：Node.js + Fastify + Prisma + PostgreSQL - 测试：Vitest + Testing Library 编码规范： - 使用函数式组件和 hooks，不使用 class 组件 - 优先使用 const 和箭头函数 - 所有函数必须有 TypeScript 类型标注 - 错误处理使用 Result 模式而非 try-catch - 组件文件名使用 PascalCase，工具函数使用 camelCase 回答规范： - 代码注释使用中文 - 变量命名使用英文 - 回答时先解释思路，再给代码 - 始终考虑性能和可访问性
效果: 设置好 .cursorrules 后，AI 的每次回答都会遵循你的技术栈和编码规范，无需每次重复说明。团队共享此文件可以统一 AI 辅助编码的质量。
📦

.cursorignore — 控制 AI 可见范围

高级

类似 .gitignore，.cursorignore 让你控制哪些文件不被 AI 索引和引用：

# .cursorignore # 排除敏感文件 .env* **/secrets/** **/credentials/** # 排除大型生成文件（减少噪声） **/dist/** **/node_modules/** **/*.min.js # 排除不相关的资源 **/assets/images/** **/*.png **/*.svg
安全提醒: 务必将包含密钥、token、密码的文件加入 .cursorignore。虽然 Cursor 不会主动泄露这些信息，但减少暴露面始终是好习惯。

模型选择策略

高级

不同模型适合不同场景，合理选择可以同时优化速度和质量：

Claude Sonnet — 日常编码、快速迭代、代码补全 速度快性价比高
Claude Opus / GPT-4o — 复杂架构设计、大规模重构 质量高推理强
Cursor Small — Tab 补全、简单修改 超快省额度
实用建议: 80% 的日常编码用 Sonnet 就足够了。只在遇到复杂逻辑设计、棘手 bug 或系统级重构时切换到 Opus/GPT-4o。

自定义 Prompt 模板

专家

为常用场景预设 prompt 模板，存储在项目的 .cursor/prompts/ 目录中：

# .cursor/prompts/code-review.md ## 代码审查模板 请对以下代码进行审查，关注以下方面： 1. 正确性 - 逻辑是否正确，是否有 bug 2. 安全性 - 是否有 XSS/SQL 注入/CSRF 等风险 3. 性能 - 是否有不必要的重渲染/请求/计算 4. 可维护性 - 命名是否清晰，抽象是否合理 5. 测试覆盖 - 边界情况是否被覆盖 对每个发现的问题，请给出： - 严重程度（P0-P3） - 具体位置（文件名 + 行号） - 修复建议（附代码示例）

使用时直接在聊天中引用模板文件即可复用标准化的审查流程。

Cursor Settings 深度调优

高级

几个关键设置能显著改善体验：

Always Search (Codebase Indexing): 确保开启代码库索引，首次索引可能需要几分钟，之后 @Codebase 搜索会非常精准。

Long Context Mode: 处理大文件时开启此选项，让 AI 能看到更多内容。但注意这会消耗更多 token。

Auto-Import Suggestions: 开启后 AI 生成的代码会自动处理 import 语句，减少手动添加依赖的麻烦。

Copilot++ 增强模式：如果你是 Pro 用户，开启后 Tab 补全的质量会大幅提升，尤其在跨函数和跨文件的上下文理解方面。

AI 编程的心智模型

专家

技术技巧之上，更重要的是建立正确的心智模型：

AI 是初级工程师，你是 Tech Lead: AI 写代码很快，但它不了解业务上下文和系统全貌。你的职责是制定方向、审查质量、把关关键决策。

验证优于信任: AI 写的每一行代码都需要你理解。如果你读不懂 AI 生成的代码，让它解释——这不仅帮助你验证，也是学习的机会。

80/20 原则: 让 AI 处理 80% 的模式化代码（CRUD、样板代码、测试），你把精力放在 20% 的核心设计和复杂逻辑上。

持续学习: 观察 AI 给出的方案，经常会发现你不知道的库、API 或设计模式。把 AI 当作一个永远耐心的结对编程伙伴。

终极目标: 不是用 AI 取代思考，而是用 AI 放大你的思考和产出。最优秀的 AI 辅助开发者，是那些知道何时依赖 AI、何时自己动手的人。
