✅ 高效协作习惯

  • 每次改动前先让 Claude 阅读相关代码
  • 提供具体的文件路径、行号和函数名
  • 分步骤迭代，每步只聚焦一个改动
  • 描述"当前行为"和"期望行为"的差异
  • 修改后立即运行测试验证
  • 使用 CLAUDE.md 固化项目规范
  • 不确定时要求 Claude 先提问再行动
  • 用 Edit 工具做精确修改，避免整文件重写

❌ 常见低效模式

  • 用"帮我改一下""优化一下"等模糊指令
  • 一次性要求实现多个不相关的功能
  • 跳过阅读代码库直接开始编写
  • 报错后不提供错误信息就要求重试
  • 忽略类型检查和测试结果
  • 不给出上下文就假设 AI 理解项目
  • 堆积多个改动后才验证
  • 每次新会话都重复解释项目背景

💡 进阶技巧

  • 先让 Claude 输出实现计划，确认后再编码
  • 复杂任务使用 /autoplan 自动生成方案
  • 利用 Memory Files 跨会话保持上下文
  • 用 Grep 搜索相似实现作为参考模式
  • 要求 Claude 解释设计决策和权衡
  • 使用 .claudeignore 排除干扰文件

🔄 RACE 循环速记

  • R — Read：先读代码，理解现有架构
  • A — Ask：有疑问先提问，不要猜测
  • C — Code：编写代码，优先用 Edit
  • E — Evaluate：跑测试，验证改动
  • 循环往复，每轮改进一点
  • 验证失败 → 回到 R 分析根因

🛠️ 工具选择速记

  • Read → 查看文件内容和代码逻辑
  • Glob → 按名称模式找文件
  • Grep → 在代码中搜索文本
  • LSP → 查看类型定义和引用
  • Edit → 修改现有文件（首选）
  • Write → 创建全新文件
  • Bash → 运行命令和测试

📐 提示词结构模板

  • 目标：我想要实现 [具体功能]
  • 文件：涉及文件 [路径列表]
  • 上下文：技术栈 [框架/版本]
  • 参考：请参考 [现有代码]的风格
  • 约束：需要满足 [具体条件]
  • 验证：完成后运行 [测试命令]
